# Lanhu Context MCP: 把蓝湖设计稿整理成 AI 可直接消费的实现上下文

## 为什么会有 Lanhu Context MCP？

蓝湖设计稿一直在很多团队里被广泛使用。蓝湖本身也有直接转代码的功能，效果并不差，但实际用下来还是会遇到几个比较现实的问题：

- 目前没有 Tailwind 支持
- `class` 命名不够语义化，类似 `text-1`、`text-2` 这类名字，后期不好维护
- 转码质量比较吃设计稿规范；稿子越规范，结果通常越稳定
- 蓝湖还要兼容 PS、Figma、Sketch 等多个设计来源，底层实现本身就比较复杂，转码结果难免会有误差
- 转码后有些最外层容器高度会被定死，对响应式布局不太友好
- 切图下载、命名和归档仍然比较手工

那在 AI 时代，这些问题能不能换一种方式解决？

`Lanhu Context MCP` 就是顺着这个想法做出来的。它想做的不是直接替你生成最终业务代码，而是先把蓝湖设计稿整理成更适合 AI 消费的上下文，补上结构化信息、切图下载处理和实现提示，再把后续工程实现交给 AI 去完成。

它不是直接承诺生成最终业务代码，而是先把蓝湖单张设计稿整理成一份更适合下游 AI 消费的实现上下文。对这个项目来说，这份上下文主要包括：

- **`HTML+CSS` 设计规格**：作为后续实现的主参考源，尽量保留设计稿里的结构、尺寸和样式细节
- **切图下载处理**：提供切图资源的原始地址、本地目标路径和下载命令，便于 AI 批量下载到项目目录，减少用户手动逐个处理资源
- **`Design Tokens`**：补充渐变、阴影、圆角等在主结构化规格之外仍有参考价值的视觉信息
- **设计图预览**：作为最终视觉对照，方便下游模型做结果校验
- **一段给下游模型使用的实现指引**：明确后续实现时的优先级、适配方式和约束条件

## 只靠 MCP 还不够

如果只有一个 MCP，其实这条链路还没有完全打通。

蓝湖很多时候需要先进入设计稿详情页，才能复制到可用链接；而复制出来的链接有时还会缺少关键的 `tid` 参数。这样一来，AI 还没开始实现，输入就已经不完整了。

所以后面又延伸出了 <a href="/ecosystem/lanhu-helper" target="_blank" rel="noreferrer">Lanhu Helper</a> 这个谷歌插件。它做的事情并不复杂，但很实用：更直接地复制可用链接和示例提示词，减少手动找参数、补参数的步骤，也让“蓝湖设计稿 -> MCP -> AI” 这条工作流衔接得更顺。

这块处理思路其实也参考了 `Figma MCP`：不仅要处理设计稿本身，还要尽量把进入 AI 之前的那一段工作流一起整理顺。只要输入链路里还有明显断点，MCP 的价值就很难真正发挥出来。

## 对比 Figma MCP

如果把 `Lanhu Context MCP` 放到更大的设计稿到代码语境里看，最容易被拿来对比的，通常就是 `Figma MCP`。

两者当然有差异，但更值得关注的不是“谁天然更强”，而是它们在工作流上的核心思路其实非常接近。

对很多团队来说，问题也不只是技术选型本身，还包括现实成本：

- Figma 对一部分团队来说有额外的授权成本
- 设计团队原本就在蓝湖协作，强行切到 Figma 或 Sketch，本身就有迁移成本
- 即便最终还是交给 AI 实现，中间多一层设计工具迁移，也不一定更省事

所以 `Lanhu Context MCP` 想解决的，并不是“证明自己比 Figma MCP 更强”，而是补上蓝湖这条链路里原本缺失的那一段。

如果只看本质，两者其实都不是“直接产出最终业务代码的终点”，而是在做同一件事：

**先把设计稿转换成 AI 更容易消费的实现上下文，再把最终落地交给 AI 来完成。**

| 维度                 | Figma MCP                                   | Lanhu Context MCP                                                           | 说明                                     |
| -------------------- | ------------------------------------------- | --------------------------------------------------------------------------- | ---------------------------------------- |
| 设计源               | Figma 设计稿                                | 蓝湖设计稿                                                                  |                                          |
| 交给 AI 的中间上下文 | React-like 结构、Tailwind、提示词、截图预览 | `HTML+CSS / HTML+Tailwind`、切图下载处理、`Design Tokens`、提示词、截图预览 | 蓝湖这边额外提供了切图下载处理和下载命令 |
| 目标                 | 降低 AI 对设计稿的猜测成本                  | 降低 AI 对设计稿的猜测成本                                                  |                                          |
| 最终落地             | 仍然要靠 AI 结合仓库去收敛                  | 仍然要靠 AI 结合仓库去收敛                                                  |                                          |

所以真正直接影响结果的，通常是这两个变量：

1. 设计稿被转换成中间上下文时，质量够不够稳定
2. 下游 AI 本身的理解、判断和实现能力够不够强

换句话说，设计稿转代码这件事里，MCP 负责的是“把输入整理得更像输入”，而最终落地效果，依然高度依赖 AI 的能力。

其中第二个变量，很多时候更关键。

## 模型能力仍然很关键

这个 MCP 给的是一份结构化设计上下文，不是最终业务代码，所以模型能力越强，后续实现通常越稳。

更强的模型通常更容易做到这几件事：

- 把切图和类名改成更像业务语义的命名
- 把设计稿里的固定高度、僵硬布局调整成更合理的页面结构
- 结合预览图做二次修正，而不是机械照抄

::: tip
如果你在 `Codex` 里做这类任务，对比过几个模型和推理档位之后，体感强推理档位通常会更稳，尤其适合生成后再做一轮视觉复核。
:::

## Tailwind

前面提到，蓝湖原生转码目前没有 Tailwind 支持，这也是 `Lanhu Context MCP` 想补上的一块。

这里要强调一点：这块并不是让 AI 临时把 `HTML+CSS` 现场改写成 Tailwind，而是先由工具侧做一轮转换，再把结果交给 AI 继续实现。

如果你的项目本身就是 Tailwind，可以直接开启 Tailwind 输出：

`--tailwindcss`

这样做的好处也很直接：交给 AI 的中间结果会更贴近项目技术栈，模型不需要先做一轮大改写，后续收敛通常会更顺。

当然，`HTML+Tailwind` 仍然只是中间结果，不等于最终业务组件。

## 更合理的预期

蓝湖设计稿转代码，本来就不是一个完美输入源。

即便有了 MCP，它也不应该被理解成“一次调用就结束”的最终代码生成器。

它真正解决的是：

- 把原本靠截图猜的部分，变成结构化规格
- 把原本手工找参数、找切图、拼链接的过程自动化
- 把 AI 的主要精力，从“猜设计”转移到“适配你的项目”

更合理的预期是：

它不能替代最终落地阶段的判断，但能把原本大量重复、低价值的手工纠偏工作压缩掉。

所以更好的使用心态是：

**把它当成一个把设计稿实现这件事往前推进很多步的工具，而不是一步到位的终点。**
